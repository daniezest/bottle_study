1. Routing & Dispatch

Lesson: When a system has a common fast case and a rare general case, give each its own code path instead of forcing everything through one uniform (and therefore slower) mechanism.
▸bottle.py:357-360 (static add) ▸bottle.py:430-446 (match) ▸bottle.py:405-415 (_compile) ▸bottle.py:278 (group limit)

if method in self.static and path in self.static[method]:

    return target, getargs(path) if getargs else {}
elif method in self.dyna_regexes:
    for combined, rules in self.dyna_regexes[method]:
        ...

Why it's good

Most routes (/about, /api/users) have no wildcards. Bottle detects this at registration time and stores them in a plain dict — matching is a hash lookup, O(1) regardless of route count. Only genuinely dynamic routes pay for regex matching, and even those are batched into combined patterns of up to 99 alternatives (the hard CPython regex-group ceiling, documented right in the code) using match.lastindex to identify which one fired. The naive version — a linear scan of individually compiled regexes for every request — would degrade as routes grow, even for apps that are mostly static.

The tradeoff

Two code paths to debug instead of one. The hand-rolled tokenizer regex parsing <name:filter:config> syntax is dense and fragile compared to a real parser — functional, but not something you'd modify without a test suite next to it.

Ask yourself in review: Is there a hot/common case buried inside a general-case code path that could be short-circuited instead?

2. The Decorator-Based API

Lesson: A registration step and a behavior-modification step are different concerns — keep the decorator that registers something dumb, and defer anything that changes behavior to a later, separate stage.
▸bottle.py:883-896 (route decorator) ▸bottle.py:497-501 (Route.call) ▸bottle.py:523-532 (_make_callback) 
def decorator(callback):
    if isinstance(callback, str):
        callback = load(callback)
    for rule in makelist(path) or yieldroutes(callback):
        for verb in makelist(method):
            verb = verb.upper()
            route = Route(self, rule, verb, callback, ...)
            self.add_route(route)
    return callback          # <-- unmodified
return decorator(callback) if callback else decorator
@cached_property
def call(self):
    """ The route callback with all plugins applied... created on demand
        and then cached to speed up subsequent requests."""
    return self._make_callback()

Why it's good

@app.route(...) returns the exact function object you defined — no wrapping, no identity change. All behavior-changing work (JSON serialization, template rendering, auth plugins) happens later, lazily, the first time route.call is accessed, and the composed result is cached so it isn't rebuilt per request. A decorator that wraps immediately would force you to know the full plugin chain at decoration time, before every plugin is necessarily installed, and would make the original callback unrecoverable for introspection.

The tradeoff

Because the decorator does almost nothing, you can't tell what a route actually does just by reading its definition — you need to separately know which plugins are installed app-wide and per-route (apply=/skip=). Discoverability is traded for composability.

Ask yourself in review: Does this decorator/registration step do only registration, or has behavior-changing logic crept in — and would deferring it make the system more composable?

3. Request/Response Object Design

Lesson: You can give callers a variable that feels like a plain global without cross-request corruption, by backing it with thread-local storage plus an explicit per-request reset.
▸bottle.py:1866-1891 (_local_property, LocalRequest) ▸bottle.py:948-949 (bind on dispatch) ▸bottle.py:168-183 (DictProperty) 
def _local_property():
    ls = threading.local()
    def fget(_):
        try: return ls.var
        except AttributeError:
            raise RuntimeError("Request context not initialized.")
    def fset(_, value): ls.var = value
    return property(fget, fset, fdel, 'Thread-local property')

class LocalRequest(BaseRequest):
    bind = BaseRequest.__init__
    environ = _local_property()
request.bind(environ)     # inside Bottle._handle, once per incoming request
response.bind()

Why it's good

Handlers just write from bottle import request and call request.GET without threading a request object through every signature — it feels like a shared global. Underneath, environ is a threading.local() slot, so each concurrent thread sees its own value, and _handle re-binds it at the start of every request so nothing leaks between requests on the same thread. DictProperty layers lazy caching on top — request.GET/.forms/.json are parsed at most once per request, cached inside that same per-request environ dict, no separate invalidation logic needed. A single mutable module-level object shared across threads would let concurrent requests overwrite each other's data.

The tradeoff

threading.local() is tied to OS threads — it doesn't compose with asyncio tasks or greenlets that don't map 1:1 to threads (see Topic 7). It also makes request/response awkward to use outside an actual WSGI call: accessing request.environ before bind() raises RuntimeError, so unit-testing a handler directly requires manually binding a fake environ first.

Ask yourself in review: Is this "global" actually a global, or context-scoped state wearing a global-looking API — and if the latter, what's the reset boundary?

4. Plugin Architecture

Lesson: Draw one explicit, narrow line between what's wired directly into the core dispatch loop and what's an opt-in, duck-typed extension point — don't let the two blur.
▸bottle.py:628-631 (core plugins) ▸bottle.py:778-788 (install) ▸bottle.py:523-532 (_make_callback) ▸bottle.py:1955-1996 (JSONPlugin) ▸bottle.py:~2020 (maintainer's TODO) 
# Core plugins
self.plugins = []
self.install(JSONPlugin())
self.install(TemplatePlugin())
def install(self, plugin):
    if hasattr(plugin, 'setup'): plugin.setup(self)
    if not callable(plugin) and not hasattr(plugin, 'apply'):
        raise TypeError("Plugins must be callable or implement .apply()")
    self.plugins.append(plugin)
    self.reset()
    return plugin

Why it's good

The "auto-serialize a returned dict to JSON" behavior feels like core functionality, but it's an ordinary plugin (JSONPlugin) that just happens to be pre-installed in __init__ — the comment # Core plugins is the only thing marking the boundary, and it's fully uninstallable. Meanwhile before_request/after_request hooks are not plugins at all — they're hardwired calls inside _handle itself. This means the plugin mechanism only needs to support one shape of extension ("wraps a callback") via loose duck typing, rather than one API trying to handle both per-route wrapping and lifecycle events.

The tradeoff

Duck typing means no enforced contract — a plugin's api = 2 attribute is a version marker nobody actually checks. Because Route.call is a cached_property, installing a plugin after routes exist requires install() to remember self.reset() to invalidate every route's cached callback — an easy gotcha, and the maintainer's own code has an unresolved # TODO: Find a better place comment acknowledging the boundary isn't perfectly clean.

Ask yourself in review: Where exactly does this system draw the line between "always runs" and "opt-in," and is that line enforced by a contract or just by convention?

5. The "Single File" Decision

Lesson: Single-file is legitimate architecture when portability is the actual product requirement — and it only stays readable if something acts like module boundaries even though the language isn't enforcing any.
▸bottle.py:4-8 (docstring) ▸232 Routing ▸579 Application Object ▸1109 HTTP/WSGI Tools ▸1947 Plugins ▸2063 Common Utilities ▸3366 Server Adapter ▸3950 Template Adapters ▸4442 Constants and Globals 
"""
Bottle is a fast and simple micro-framework for small web applications. It
offers request dispatching (Routes) with URL parameter support, templates,
a built-in HTTP Server and adapters for many third party WSGI/HTTP-server and
template engines - all in a single file and with no dependencies other than the
Python Standard Library.
"""

Why it's good

The stated goal — zero dependencies beyond the standard library, drop-in portability — is a real product requirement for a micro-framework, not laziness. 4,583 lines are organized into 15 banner-delimited sections ordered by dependency: helpers → routing → app object → request/response → plugins → utilities → server adapters → template adapters → globals. Each section reads top-to-bottom on the assumption that later sections may depend on earlier ones, never the reverse — a manual, convention-enforced stand-in for what package imports would give for free.

The tradeoff

There's no actual enforcement. Nothing stops code in a later section from reaching backward into an earlier section's internals — the ordering discipline is a convention the author maintains by hand. This only stays workable because Bottle is small, slow-moving, and single-maintainer; the same approach on a fast-moving multi-contributor project would rot without a linter enforcing dependency direction.

Ask yourself in review: If this were split into real modules, would the import graph be a clean DAG — and if the single-file version doesn't obviously map to one, is a boundary already being violated?

6. Error Handling & Failure Modes

Lesson: Model errors as ordinary values flowing through your normal output path, not as an out-of-band control-flow exception special-cased at every call site.
▸bottle.py:460-464 (404/405) ▸bottle.py:960-980 (_handle exceptions) ▸bottle.py:1009-1016 (_cast dispatch) ▸bottle.py:1076-1089 (wsgi outer catch) ▸bottle.py:2714 (abort) ▸bottle.py:918-938 (error decorator) ▸bottle.py:4447 (DEBUG global) 
if allowed:
    raise HTTPError(405, "Method not allowed.", Allow=allow_header)
raise HTTPError(404, "Not found: " + repr(path))
if isinstance(out, HTTPError):
    out.apply(response)
    out = self.error_handler.get(out.status_code, self.default_error_handler)(out)
    return self._cast(out)
if isinstance(out, HTTPResponse):
    out.apply(response)
    return self._cast(out.body)

Why it's good

HTTPError is a subclass of HTTPResponse, itself a normal response object. A 404 raised deep inside Router.match() is caught as a control-flow exception but immediately handled as data — it flows into _cast(), the exact same function that turns a handler's returned string or dict into a WSGI response body. That's why abort(404, "msg") is a one-liner, and a custom 404 page is just a dict assignment (error_handler[404] = callback), not a special dispatch mechanism. Checking "did the router find a route?" with an explicit if/else at every call site would scatter this logic everywhere instead of centralizing it in one type-dispatch.

The tradeoff

Because uncaught real exceptions are also converted into HTTPError(500, ...) at the same layer, and there's a second outer catch-all in wsgi(), a genuine bug (a stray NameError) is silently swallowed into a 500 response unless catchall is turned off — the classic "caught everywhere, debugged nowhere" failure mode of broad except Exception. Correctness depends on the operator remembering to flip debug()/catchall per environment.

Ask yourself in review: Is this error a value flowing through the normal path, or an exception needing a bespoke handler everywhere it might occur — and is a broad except Exception here hiding real bugs behind a generic response?

7. Where Bottle Says No

Lesson: Restraint means shipping a real default for the common case and pushing the uncommon case to an explicit, swappable adapter — instead of one abstraction trying to serve both.
▸bottle.py:4-8 (docstring) ▸bottle.py:4112 (SimpleTemplate) ▸bottle.py:4083 (Jinja2Template) ▸bottle.py:4040 (MakoTemplate) ▸bottle.py:3688 (AiohttpServer)

No inline snippet here — the point is structural (what's built-in vs. adapter), not one quotable block. Click through to compare SimpleTemplate against Jinja2Template/MakoTemplate side by side.

Why it's good

Bottle ships a genuinely usable minimal template engine (SimpleTemplate) so a project has zero external dependencies to render its default error page or basic views — but Jinja2/Mako/Cheetah exist as separate adapter classes you opt into, not as a unifying templating abstraction Bottle tries to own. Same pattern for concurrency: there's no async def handler support anywhere in the core request cycle; async servers like aiohttp/gevent/eventlet are wired in only as ServerAdapter subclasses that still present the same synchronous WSGI contract to your route functions. There's also no ORM and no CSRF protection anywhere in the file — both are "bring your own," full stop. A "bigger" framework would likely try to unify sync/async or unify templating engines; Bottle picked one real default and made everything else pluggable.

The tradeoff

Anyone who needs async handlers, an ORM, or CSRF protection gets no help from the framework at all — not a partial abstraction to build on, just nothing. That's the point of staying a micro-framework, but it means Bottle is a poor fit once a project's needs outgrow its narrow default, with no gradual on-ramp.

Ask yourself in review: Is this abstraction trying to unify two fundamentally different cases, and would a hard default plus an explicit adapter be more honest than a leaky unifying interface?

Highest-Leverage Takeaways

Ranked by how often you'll see the pattern reused outside this codebase:

  1. Split the fast/common path from the slow/general path instead of unifying them (Topic 1) — cache-then-fallback, static-then-dynamic, happy-path-then-exception-path all share this shape.
  2. Decouple registration from behavior (Topic 2) — decorators/registries that return the original object and defer wrapping to a later composed stage show up in DI containers, ORM model registries, test fixtures.
  3. Model errors as values flowing through the normal path, not a parallel control-flow system (Topic 6) — the same insight behind Result/Either types in other languages.
  4. Context-scoped state that feels global but isn't (Topic 3) — thread-locals, contextvars, React context, Flask's own request context all solve this the same way.
  5. Restraint as "own the default, adapt the edge case" (Topic 7) — the rarest to execute well, and the highest-value judgment call in review: does this new abstraction genuinely serve two needs, or quietly worsen the common case to accommodate a rare one?
